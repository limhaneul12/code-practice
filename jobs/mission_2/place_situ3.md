# ② 프롬프트 진단 설계 — "설계했으나 A·B·퀄리티 모두 실패" 케이스

## 1. 케이스 정의
- 학습자가 원하는 결과(A)를 의도하고 프롬프트를 나름 설계함
- 결과(B)가 의도와 다르게 나옴
- 나온 결과(B)마저 퀄리티가 낮음
- 특징: **이미 프롬프트 작성을 시도한 상태**이며, 방향 실패와 완성도 실패가 중첩됨

## 2. 진단 원칙
| 원칙 | 근거 |
|---|---|
| 의도 → 넣은 것 → 간극 순서로 진단 | 이미 시도한 학습자이므로 누락 지적보다 본인 산출물을 먼저 꺼내게 함 |
| 방향 문제와 완성도 문제 분리 | 뭉쳐서 진단하면 "전부 실패" 좌절이 확대됨 |
| 수정은 한 번에 하나만 | 결과 변화의 인과를 학습자가 체감 |
| 판단 기준은 기획(메시지·타겟)에 고정 | "예쁨"이 아닌 "의도 부합" 기준 유지 |

# 3. 진단 흐름

## 3.1 진입
- 질문: "프롬프트 넣었을 때 원하던 게 나왔어요?"
- 조건: "원한 것도 아니고 결과도 별로다" 응답 시 본 흐름 진입

## 3.2 1단계 — 의도 확인 (P1)
- 질문: "원래 뭐가 나오길 바랐어요? 말로 설명해볼래요?"
- 분기
  - 의도 설명 가능 → 2단계
  - 의도 불명확 → 분기 A

**분기 A (의도 미확립)**
- 질문: "이 장면이 광고에서 뭘 보여주는 컷이었죠? 정한 메시지와 연결하면?"
- 처리: ①기획으로 회귀해 메시지에서 의도 재도출 → 확립 시 2단계

## 3.3 2단계 — 작성 내용 확인 (P2)
- 질문: "그거 나오게 하려고 프롬프트에 뭐라고 썼어요? 그대로 읽어주세요."
- 목적: 실제 프롬프트를 소리 내어 읽게 하여 학습자가 스스로 부족함을 인지

## 3.4 3단계 — 간극 진단 (P3)
- 질문: "바란 것(말)과 쓴 것(프롬프트)을 비교하면, 말로는 했는데 프롬프트엔 안 들어간 부분이 있죠?"
- 갈래
  - **방향 문제**: 의도 요소가 프롬프트에 누락
    - 예: "'따뜻한 새벽 감성'이라 했는데 프롬프트엔 '아침 카페'만 있다. AI가 '따뜻함'을 어떻게 아나?"
    - 처리: 누락된 의도 요소를 언어화하여 추가
  - **완성도 문제**: 방향은 맞으나 퀄리티 부족
    - 질문: "방향은 맞다. 화질·디테일이 아쉬운 거면 다른 문제다. 품질·조명·구체 묘사 표현이 있나?"
    - 처리: 방향과 완성도를 분리해 별도 진단

## 3.5 4단계 — 수정 및 재시도
- 지시: "찾은 것 하나만 프롬프트에 더해 다시 돌려보자."
- 원칙: 한 번에 하나씩 수정
- 사후 확인: "이번엔 뭐가 달라졌어요?" (변화를 학습자가 직접 확인)

## 3.6 공통 — "모르겠다" 대응
- 적용: 어느 단계든 판단 정지 시
- 처리: 가장 작은 실험 하나로 축소
  - 예: "'새벽 감성' 이 표현만 맨 앞에 넣고 다시 돌려보자."
- 목적: "프롬프트를 바꾸면 결과가 바뀐다"는 인과 체감
